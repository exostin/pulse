# AI System Orchestration: Control Loops, Evaluation, and Reliability for LLM-Based Systems

A working reference to the named parts of control theory, automated QA, and SRE — translated for solo operators of LLM-based systems. The vocabulary you need to think about your AI work as a system being controlled, not as a collection of prompts being tweaked.

Generated via [[working-reference-template]] on 2026-05-08. The filled prompt is preserved at the bottom of this doc as provenance.

## The taxonomy

Six layers stacked, plus a cross-cutting layer that runs through all of them:

1. **Telemetry** — what you record at all
2. **Evaluation** — how you grade behavior against intent
3. **Control** — how a measured gap turns into an action
4. **Intervention** — the things you actually change in response
5. **Process** — how you and the dyad learn from incidents
6. **Failure modes** — first-class layer, named so you can reach for them as defenses

The layers are roughly bottom-up: telemetry feeds evaluation, evaluation feeds control, control selects interventions, process turns recurring failures into structural fixes, and the failure-mode names live at every level as a working differential diagnosis.

## 1. Telemetry — what you actually see

[Decision trace] — load-bearing  
The single most useful kind of logging in an LLM system. Not "what did the model say" but "what was the model handed, what intermediate decisions did it make, and which option did it pick from the choices it had." PULSE's Session Log is exactly this: not a record of work output, but a record of *why* things were ranked, suppressed, classified the way they were. Decision traces are what let you do post-hoc analysis when behavior drifts — without them, you can only re-run the system and hope it misbehaves again. The classic mistake: logging only inputs and outputs and assuming you can reconstruct the middle. You can't, because LLM internal state isn't recoverable from input/output pairs. Contrast with conventional application logging, which is mostly oriented toward "did this code path execute" — for LLM systems you want "what was the model's epistemic position when it made this choice."

[Trace, span, event] — load-bearing  
Borrowed from distributed-systems observability and worth keeping the names. A *trace* is one end-to-end flow (a /pulse run, an agent dispatch, a multi-tool sequence). A *span* is one bounded sub-operation within a trace (a tool call, a sub-agent invocation, a single prompt-completion exchange). An *event* is a point-in-time annotation on a span (a retry happened, a guardrail triggered, a fallback activated). The unlock: when something goes wrong, you want to navigate from "the run misbehaved" → "in this span" → "at this event." Without that structure you're scrolling logs by timestamp and hoping. The analogy to OpenTelemetry is honest, but watch out: in distributed systems spans are usually services; for LLM systems they're cognitive steps. The right granularity is one span per "decision the model or the orchestrator made," not one span per network call.

[Golden signals] — load-bearing  
The SRE shorthand for "the four things you almost always want graphed first": latency, traffic, errors, saturation. For an AI system the translation isn't 1:1, but the *idea* is: pick three to five signals that, if all four look fine, the system is probably fine, and if any look bad, you'll know roughly where to dig. For LLM systems the candidates are: response latency, cost per task, eval pass rate against a held-out set, hallucination/refusal rate, and some measure of distribution shift in outputs over time. Don't get fancy first — pick a small, stable set and watch them. The temptation is to instrument everything; the discipline is to pick few enough signals that you'll actually look at them daily.

[Cardinality] — adjacent  
The number of distinct values a logged dimension can take. If you log `model_version` you have low cardinality (a handful). If you log `user_message` you have effectively infinite cardinality, and your storage and aggregation costs explode. Worth knowing because you'll be tempted to log every prompt and answer; the right move is to log structured metadata about prompts (length, model, route taken, tools used) and store the raw text in a separate, less-frequently-queried place. The pitfall: high-cardinality fields silently destroy your ability to aggregate — a chart that says "errors over time" stops being useful when each error is unique.

[Per-token vs per-request observability] — adjacent  
LLM systems have a level of granularity below "request" — token streams, log probabilities, cache hit ratios, time-to-first-token. Most personal-scale systems don't need this. Reach for it when you're debugging latency-sensitive workflows, when you're trying to understand why a model produced a particular continuation, or when prompt caching is a meaningful fraction of your cost.

[Sampling] — adjacent  
You don't log everything. You log a sample, weighted toward what you want to study. Two flavors that matter. *Head-based sampling*: decide at trace start whether to keep this trace, based on a coin flip. *Tail-based sampling*: keep all traces briefly, then decide what to keep based on what happened (always keep errors, sample successes). Tail-based is usually right for AI systems because the interesting traces are rare and you want them all. The failure mode: head-based sampling at low rates means you almost never see your failure traces.

## 2. Evaluation — grading behavior against intent

[Eval] — load-bearing  
A test for an LLM system. The defining property: an eval has an *expected* behavior or quality, and a *grader* that decides whether the actual behavior matched. Both pieces are non-trivial. "Expected" is rarely a single string — usually it's a property the response should satisfy. The grader is often itself an LLM, which means your grader can be wrong, biased, sycophantic, or drift over time. The skill is in defining evals that are stable — that don't reward shallow matches and don't punish acceptable variations. *Most evals are bad evals*: they over-specify form ("must contain the word 'sorry'") and under-specify substance ("acknowledges the user's concern"), so they pass things that shouldn't and fail things that should. The two questions to ask of any eval: (1) if I tweak the response in a way that doesn't change its substance, does the eval still pass? (2) if I tweak the response in a way that does change its substance, does the eval catch it? An eval that fails either is mostly noise.

[Offline eval vs online eval] — load-bearing  
*Offline*: you run the system against a fixed dataset of inputs and grade the outputs. Cheap, repeatable, lets you compare versions head-to-head. *Online*: you grade behavior on actual live traffic — sampled, often via LLM-as-judge or human spot-check. More expensive and slower, but it's the only way to detect drift between your eval set and the real world. Build offline first; you need it to make any change with confidence. Add online when you start trusting the system enough to leave it alone for stretches. The trap is using only one: offline-only systems overfit to the eval set and silently break on real usage, online-only systems can't safely iterate because you have no way to compare versions.

[Golden set / regression suite] — load-bearing  
A specific, curated set of cases where you know what should happen. Two sub-flavors that earn their own names. *Regression set*: cases the system handles correctly today, which you check after every change to confirm you didn't break anything. *Stretch set*: cases the system currently handles poorly that you're trying to improve. Both belong in the suite. The regression set is what protects you from making things worse; the stretch set is what makes "did this change help?" a real question instead of a vibe. The single most undervalued artifact in personal AI work — it costs little to maintain and pays back enormously, especially because the cases that you want in it are exactly the ones you've already debugged once and don't want to re-debug. The discipline is *adding to the set when you discover a new failure*, not just when you set out to build the set.

[LLM-as-judge] — load-bearing  
Using one LLM to grade another's output. Indispensable because most properties you care about are too fuzzy for string matching and too expensive for human review at scale. Three things bite people. First, judges are *biased* — they prefer longer responses, responses in their own style, responses that match the judge's training distribution. Second, judges are *not calibrated* — a "yes this is good" doesn't mean what you think it means until you've checked. Third, if you use the same model family as judge and judged, you have a *correlated-error* problem: both might be wrong in the same way, so the judge endorses the bad output. The defense: spot-check judge outputs against your own assessment, regularly, especially when scores trend in surprising directions. A useful rule: when the judge says something improved, ask yourself if you'd reach the same conclusion looking at the same examples. If you find yourself going "well, it did say it was better," that's the moment of correlated error.

[Pairwise vs absolute scoring] — load-bearing  
When you ask an LLM (or a human) to evaluate, you can either ask "rate this 1-5" (absolute) or "is A or B better?" (pairwise). Pairwise is dramatically more reliable. Absolute scores drift, anchor on the recent past, and compress to the middle of the scale. Pairwise comparisons stay stable across sessions. The cost: ranking N items requires roughly N log N comparisons, vs N for absolute scoring. Worth it for important rankings. The contrast that's easy to miss: pairwise tells you which of two is better but not how good either is, and so doesn't catch the case where everything is bad. Pair it with a periodic absolute "is this acceptable at all" check.

[Calibration] — load-bearing  
The match between confidence and accuracy. A well-calibrated system that says "80% sure" is right about 80% of the time. LLMs are usually badly calibrated — overconfident on hard questions, sometimes underconfident on easy ones, and especially miscalibrated near the boundary of their training data. PULSE's priority calibration log is calibration in this sense: a record of where the system's surfacing predictions diverged from what mattered, used to correct future predictions. The general technique: keep score on your system's own confidence, and apply a correction when confidence and outcome systematically diverge. Two traps. First, calibrating on small samples — you need enough cases for the correction to be real and not noise, and the correction itself can be wildly off if your sample is small. Second, *over-correction*: if you correct each error fully, you'll oscillate as new errors come in. Move slowly toward the implied correction (low gain, in control terms).

[Eval rot, overfitting to eval] — load-bearing  
The eval set, once it exists, becomes a target. You will, consciously or not, tune the system to pass your evals — and that's the point. The danger is when the eval set stops representing real usage, either because real usage drifted or because you've been gaming the eval without realizing. Two defenses worth practicing. First, *rotate cases* in and out of your eval set so you don't hardcode to specific examples. Second, keep a *blind set* — a held-out group of cases you don't look at while developing and that you only sample from periodically. If blind-set scores diverge from regular-eval scores, you're overfitting. The blind set has to actually stay blind; if you "just check it" frequently, you've lost the property. The discipline is hard because it feels wasteful to have evals you don't use much, but the value is exactly in their unfamiliarity to the system you're tuning.

[Coverage] — adjacent  
What fraction of the behavior space your evals actually exercise. Borrowed from software testing, where it's already a problematic concept (line coverage doesn't mean behavior coverage). For LLM evals it's worse — you have effectively infinite input space, and you're sampling a tiny corner. Don't take a coverage number too seriously; do periodically ask "what kinds of cases does my eval set never test, and is that a problem?" One useful move: characterize your eval set along a few dimensions (length, domain, tone, complexity) and notice the gaps in the joint distribution.

[Property-based eval] — adjacent  
Instead of "for input X, expect output Y," you specify properties any acceptable output must satisfy ("response cites at least one of the provided sources," "response does not contain the user's name," "response under 200 words"). Useful when the space of acceptable outputs is large but the constraints are precise. The complement to golden sets: golden sets test specific cases, property tests test universal claims. The failure mode: property tests can pass because they're vacuously true on garbage outputs ("response is under 200 words" passes for a one-word non-answer). Pair them with at least one substance check.

[Snapshot test] — named-only  
*Snapshot test* — record an output once, alert when it changes. Reach for this if you want to detect any change in behavior, knowing you'll have to manually re-approve every legitimate change. Mostly a brittleness amplifier for LLM systems, but useful as a tripwire on workflows you're not actively iterating on.

[Mutation testing, fuzz, adversarial eval] — adjacent  
Adversarial methods that perturb inputs to find brittleness. Especially relevant for LLM systems because tiny prompt changes can cause large behavior changes. The simplest version is just rephrasing — does your system still do the right thing if I ask for the same thing in different words? If not, you have hidden brittleness. Worth running occasionally even on workflows you trust; the cost is low and the surprises are real. The closely related *prompt brittleness* sits in the failure-mode section.

[Inter-rater agreement] — named-only  
*Inter-rater agreement* (Cohen's kappa, Krippendorff's alpha) — formal measures of how much two graders agree once you correct for chance agreement. Reach for this when you have multiple human or LLM judges and you need to know whether your evaluation is meaningful at all. If your judges don't agree with each other, your evals don't measure anything stable.

## 3. Control — turning a measured gap into an action

[Feedback loop, open vs closed] — load-bearing  
The central concept of control theory and the central thing most AI systems lack. A *closed-loop* system measures its own output, compares it to a target, and adjusts. An *open-loop* system runs and hopes. Most personal AI work is open-loop: you set up a prompt, run it, eyeball results, change the prompt, repeat. Closing the loop means putting in place: a measurement (eval), a target (set point), an automatic comparison (delta), and a mechanism that responds (intervention). PULSE's calibration log is a partial loop closure — it measures errors and feeds them back into future ranking. Most loops you'll build will be partial; that's fine. The diagnostic question to ask of any system: where does the loop close, and where does it stay open? An open loop isn't always wrong (some things shouldn't be automated), but you should know which is which.

[Set point] — load-bearing  
The target you're trying to hold the system at. Trivial in mechanical control (the thermostat is set to 68°F). Genuinely hard in AI systems because the things you most want to control — quality, helpfulness, alignment with intent — are hard to express as a single number. You typically end up with a *vector* of set points (eval pass rate above X, hallucination rate below Y, latency under Z, cost under W) and accept that you'll trade them off. The discipline: actually write down the set points. A system without explicit set points is one where you'll keep moving the goalposts after the fact, and you won't notice yourself doing it. Even bad set points that you revise are better than no set points. The trap that catches new operators: choosing set points before you have measurements, then bending them to whatever the system happens to produce. Order matters — measure first, then set targets that mean something.

[Error signal] — load-bearing  
The gap between current state and set point. Sounds obvious. The hard part is that for AI systems, "current state" usually requires running an eval, which has cost and latency. So your error signal is sampled, not continuous, and there's a *dead time* between when behavior changes and when you notice. This shapes everything about how you can control: if error signals come once a day, your loop has a one-day lag and shouldn't oscillate faster than that. Many AI orchestration mistakes come from operators reacting to a single eval failure as if it were a continuous error signal.

[Gain, oscillation, stability] — load-bearing  
*Gain* is how aggressively you respond to an error signal. Too low and you barely correct; too high and you overcorrect, and the system swings. *Oscillation* is the failure mode of high-gain control: you push, overshoot, push back, overshoot the other way. In AI work this looks like: "the system was being too verbose, I added 'be concise' to the prompt, now it's giving one-word answers, I added 'with appropriate detail,' now it's verbose again." The cure is lower gain — smaller adjustments, with measurement between them — and patience to let the loop settle before pushing again. *Stability* is the property that the system, left alone after a small perturbation, returns to the set point rather than wandering off. Most control problems in AI systems are stability problems hiding as quality problems.

[Damping, hysteresis] — load-bearing  
Two different ways to suppress oscillation. *Damping*: respond proportionally to the rate of change as well as to the error itself, so you stop pushing as you approach the target. *Hysteresis*: build in a deadband where small errors don't trigger any response. PULSE has hysteresis in places — small weight changes don't trigger re-ranking, the suppression batch threshold has stickiness around 40%. Without it, the system would constantly re-arrange itself in response to noise. The general principle: if a signal is noisy, treat small movements as no movement. The named contrast worth keeping: damping slows your response, hysteresis ignores small responses entirely. Use damping when the signal is real but you're moving too fast; use hysteresis when the signal is mostly noise below some threshold.

[Dead time, loop latency] — load-bearing  
The delay between an action and the feedback that reveals its effect. In a thermostat, dead time is small; in an AI system it can be days (you change a prompt today, you don't know if it improved real usage until you've sampled enough sessions). A loop with long dead time should have *low gain* and slow updates — otherwise you're correcting based on outdated information. The mistake: short-loop intuitions ("I'll iterate fast") applied to long-loop processes ("the eval is from last week"). You'll oscillate, and the oscillation will look like the system being unstable, when actually you're the destabilizer. The honest acknowledgment most operators don't make: if your eval feedback takes a week, your effective iteration rate is weekly, regardless of how fast you can change the prompt.

[Observability, controllability (control-theoretic sense)] — load-bearing  
Two questions to ask of any system you want to control. *Observability*: can you actually see the state you care about, given the signals you have? If quality is your concern but you only log latency and cost, your system is unobservable for the dimension you care about. *Controllability*: can you actually affect the state, given the inputs you have? If you can change the prompt but not the model and not the data, your control space is narrow. A surprising amount of "this system is hard to improve" turns out to be one of these — you can't see what's wrong, or you can see it but you can't move it. Don't confuse the SRE/observability sense (instrumentation richness) with the control-theoretic sense (state-recoverability from outputs); they're related but not the same. The control-theoretic sense is sharper: if changing your inputs in any way produces no detectable difference in your outputs, you don't have observability.

[Bang-bang vs proportional control] — adjacent  
Bang-bang: the controller is fully on or fully off (the heater runs at full blast or not at all). Proportional: the controller's output scales with the error. Most AI interventions are accidentally bang-bang — guardrails fire or don't, fallbacks activate or don't, the model is the big one or the small one. Sometimes that's correct. Sometimes you want proportional: "if confidence is low, add a verification step; if very low, escalate to human; if extremely low, refuse and explain." Worth knowing the distinction so you can choose deliberately.

[PID controllers] — named-only  
*PID* — proportional-integral-derivative, the workhorse controller of classical control theory. P responds to current error, I to accumulated error over time, D to rate of change. For AI systems, PID is more useful as a metaphor (am I responding to current state, history, or trend?) than as a literal implementation. Reach for it if you find yourself building a controller and wondering whether to weight history or trend.

[Cybernetic loop, requisite variety] — adjacent  
Older, broader frame for the same idea, from cybernetics. The interesting thing it adds is *requisite variety* (Ashby's law): the controller must have at least as much variety as the system it controls, or it can't keep up. In LLM terms: if your control mechanism is "tweak the prompt manually once a week" and the system can fail in a thousand ways, you don't have requisite variety. That's an argument for either narrowing the failure space (guardrails, structure) or widening the controller (richer interventions, more frequent updates). Worth keeping the name because it's a sharper way to diagnose "why does this system feel out of control" than vague gestures at complexity.

## 4. Intervention — the things you actually change

[Retry with exponential backoff and jitter] — load-bearing  
The reflex move when something fails transiently. Three details to get right. *Exponential* backoff (each retry waits longer than the last) keeps you from hammering a struggling service. *Jitter* (random variation in the wait) keeps multiple retrying clients from synchronizing into a thundering herd. *A retry budget* (cap total retries, or fraction of total requests that are retries) keeps a partial outage from becoming a self-inflicted DDoS. For LLM systems, retries are also where you accidentally double-charge yourself for tokens, so track retry rate as a signal. The trap specific to LLMs: retrying on a "bad output" (rather than an error) implicitly assumes the output was unlucky rather than the prompt was wrong. Often the prompt was wrong, and three retries just spend three times as much money getting three bad outputs.

[Timeout] — load-bearing  
A bound on how long you'll wait. Easy to set badly. Two failure modes: timeouts so tight that legitimately slow responses get killed (you lose work that would have succeeded), and timeouts so loose that a stuck call holds up everything downstream. The right number is empirical — look at your latency distribution and pick something like p99 plus a margin. Then revisit when the distribution shifts. For LLM systems specifically, watch the gap between time-to-first-token and total-time; a slow first token usually means something different (queueing, cold cache) than a slow stream (long output, slow generation).

[Circuit breaker] — load-bearing  
A switch that flips open after enough failures and stops sending traffic to the failing component for a while, then tentatively allows a few requests to test recovery. The point is to fail fast when something is broken instead of piling up timeouts. For AI systems, the obvious applications are: stop hitting a model API that's returning errors; stop calling a tool that's been failing; stop running a sub-agent that's been looping. Less obvious: when your *eval scores* drop sharply, treat that as a circuit-breaker signal — pause the change that caused it and investigate before pressing forward. Circuit breakers compose nicely with canary deploys: bad canary → circuit breaker opens → traffic stays on the old version while you debug.

[Bulkhead] — adjacent  
Isolation between parts of the system so failure in one doesn't sink the whole. The naval-architecture analogy is exact. In AI systems: isolated agents that can fail without poisoning the parent context, separate budgets per workflow so one runaway doesn't consume your daily quota, separate eval suites per skill so a regression in one doesn't block work on another. The general move: ask of any failure, "what does this take down with it?" and put walls where the answer is too much.

[Backpressure] — adjacent  
A signal from a downstream component to an upstream one that says "slow down, I can't keep up." Mostly relevant for high-throughput systems, but the concept applies to agent orchestration: if a sub-agent is producing more than the parent can usefully consume, you want a way to slow it down rather than buffer indefinitely.

[Fallback, graceful degradation] — load-bearing  
What you do when the preferred path doesn't work. *Fallback*: a specific alternative ("if the main model fails, use the smaller model"). *Graceful degradation*: a general posture ("if some signals are missing, do the best we can with what we have"). Both are about avoiding all-or-nothing failure. In AI systems, the highest-leverage fallbacks are usually: smaller/faster model when the big one is slow, cached previous answer when fresh generation fails, simpler prompt when the rich one returns garbage, and "ask the human" when all of the above fail. The trap: building such smooth fallbacks that you never notice the primary path is broken. Always log fallback activations as a distinct signal — frequent fallbacks mean something is wrong even when the system "works."

[Idempotency] — load-bearing  
The property that doing something twice has the same effect as doing it once. Critical because the moment you have retries (and you will), the same operation can fire multiple times. If saving a Note isn't idempotent, you get duplicates; if a triage isn't idempotent, you re-classify things that are already classified; if a Map update isn't idempotent, you get appended duplicates of the same entry. The discipline is to make every external action idempotent by default — usually with an idempotency key (a stable identifier the operation respects). Hard to retrofit, easy to design in from the start. The deeper version: even read-only operations should be idempotent in the sense that running them twice doesn't produce different results — which sounds trivial until you realize LLM calls don't satisfy this without temperature=0 and a seed.

[Guardrail (input and output)] — load-bearing  
A check that runs before (input guardrail) or after (output guardrail) the model call. Input guardrails protect the model from bad inputs (prompt injection attempts, requests for things you've decided not to do, malformed payloads). Output guardrails protect the rest of your system from bad outputs (hallucinated tool calls, leaked private info, format violations). The trap: guardrails that aren't tested are guardrails that don't work. Treat each guardrail as a hypothesis ("this regex catches the bad case") and write evals that confirm it actually does. A surprising number of "we have safety in place" systems have guardrails that pass everything because the regex was never validated. Distinct from *intervention* in the loose sense: guardrails are gates, not corrections — they say no, they don't fix.

[Canary, progressive rollout] — adjacent  
Roll a change out to a small slice of usage first, monitor, then expand. For personal AI systems, the slice can just be "use the new prompt for one workflow this week." The discipline is to *actually compare* — you're not just trying the new thing, you're holding it next to the old thing on the same kinds of inputs and noting differences. Without comparison, "I tried it and it seems fine" is just confirmation bias. The contrast with A/B testing: canaries are about safety (catch regressions before they hit everyone), A/B is about decision (which version is actually better). They're often run together but they're different functions.

[Feature flag, kill switch] — load-bearing  
A toggle that lets you turn behavior on or off without changing code. The kill switch is the special case where the only state is "off" — used when something is misbehaving and you need it gone immediately. For personal AI systems, the equivalent is a way to disable a skill, a guardrail, or a sub-agent quickly when it starts misfiring. Build kill switches before you need them; the moment you need one is a bad time to add one. The mature version is *typed flags*: not just on/off, but "use this version of the prompt," "use this model," "skip this guardrail" — so you can configure the system declaratively rather than by code edit.

[Rate limiting, quota] — adjacent  
A cap on how often or how much. Necessary at any boundary that costs money (model APIs) or that can be abused (multi-agent loops that might recurse). Two kinds: *hard limits* (over the cap → reject) and *soft limits* (over the cap → warn or queue). Personal systems mostly need hard limits on cost-sensitive operations and soft alerts on worrying trends. The agent-orchestration-specific case worth flagging: recursive sub-agent calls without depth limits are a class of resource bug that's especially easy to hit with LLMs that don't notice they're looping.

## 5. Process — how you and the dyad learn from incidents

[Postmortem, blameless review] — load-bearing  
A structured look back at something that went wrong. The "blameless" prefix matters: the goal is to find the conditions that made the failure possible, not to assign fault. For solo work, the framing is: not "what did I do wrong" but "what about the system or the situation made this failure plausible, and what would catch it next time." The structure that works: what happened (timeline), what was the impact, what were the contributing factors (multiple, usually — single-cause stories are almost always wrong), what's changing as a result (concrete, owned, not just "be more careful"). For PULSE, the natural cases are: the system surfaced something noisy and important got buried; a `/triage` misclassified an item; a calibration adjustment overcorrected; a sub-agent loop ran longer than it should have. Each is a postmortem candidate. The discipline that's hard: writing them when nothing dramatic happened — "minor weirdness" postmortems are where you catch latent failures before they become big ones.

[Five whys] — adjacent  
A specific postmortem move: ask "why" five times in succession to push past surface causes. Useful as a tool, dangerous as a doctrine — the real causal structure is usually a tangled web, not a chain, and pretending it's a chain can hide the actual fragility. Use it as a *prompt* to keep digging, not a *method* that produces the right answer. The honest critique: people who run five-whys often end at "human error" or "process failure," neither of which is a real cause; the actual causes were upstream conditions that made the human error plausible.

[Swiss cheese model] — load-bearing  
A framing from accident research: failures usually require multiple defenses to fail at once, like holes in stacked slices of Swiss cheese aligning. The right question after an incident isn't "why did defense X fail" but "why didn't defenses Y and Z catch what X missed." The model gives you vocabulary for thinking about layered defense in a non-naive way: each layer has holes, the design question is whether the holes line up. The contrast that matters: "what was the cause" (single-link thinking) vs "which combination of failures was required" (multi-link thinking). The latter is almost always a more accurate description of what happened.

[Active vs latent failure] — load-bearing  
Twin concepts from the same accident-research tradition. *Active failures*: errors at the sharp end, the immediate trigger of a bad outcome (the operator clicked the wrong button, the agent called the wrong tool). *Latent failures*: conditions in the design or environment that made the active failure plausible or unavoidable (the buttons were placed too close together, the tool descriptions were ambiguous). The point is that fixing the active failure ("be more careful") doesn't prevent recurrence; fixing the latent conditions does. For AI systems, latent failures are usually upstream: prompt structure that invites confusion, eval that doesn't cover this case, observability gap that hid the warning signs.

[Error budget] — load-bearing  
A pre-agreed amount of failure that's acceptable. For production SRE this is precise — "99.9% uptime allows 43 minutes of downtime per month." For personal AI work the version is: "this system is allowed to surface noise N times in a week before I treat it as a quality regression." The point is to decide *in advance* what counts as too many failures. Without a budget, every individual failure feels like it should be fixed; with one, you can let small failures go and focus attention on the patterns that exhaust the budget. The deeper move: error budget *spending* is a permission slip — you can take more risk (deploy faster, try bigger changes) when you have budget left, and you should slow down when you've spent it.

[Burn rate] — adjacent  
How fast you're consuming the error budget. A high burn rate over a short window is a signal that something has materially changed for the worse. The discipline is to react to *rate of consumption*, not just absolute count — a system that's quietly using up its budget faster than usual is worth investigating before the budget is gone.

[Runbook] — load-bearing  
A written procedure for handling a specific situation. The classic SRE form is "if alert X fires, do Y." For AI systems the equivalent is a record of how you've handled known failure modes — "when the agent starts looping, do this; when calibration scores drift down, do this." The point isn't to mechanize your judgment but to externalize what you've learned, so that future-you (or future-Claude, or a teammate) doesn't have to rediscover it. PULSE's skills are runbooks in this sense — they encode "here's how to handle this kind of situation." The contrast with documentation: docs explain how the system works in steady state, runbooks explain what to do when it's not.

[Game day, chaos engineering] — adjacent  
Deliberately introduce failure in a controlled way, see how the system responds, fix what doesn't hold up. The personal-scale version is: occasionally feed your AI system inputs designed to break it (prompt injection attempts, contradictory instructions, ambiguous goals) and see what happens. Or: simulate the failure of one component (the model returns nothing, a tool returns garbage) and verify your fallbacks fire. The point is to find the brittleness *before* it finds you. The deeper culture point: chaos engineering only works in environments where breaking things is allowed; if every game-day failure is treated as an incident to be punished, no one will run them.

[Toil] — adjacent  
SRE term for repetitive, manual, automatable work that scales linearly with system size. The principle: track it, and treat reduction of toil as a first-class goal, because toil eats the time you could spend on improvement. For AI work the toil candidates are: manually re-running the same eval, manually triaging the same kinds of items, manually checking the same guardrails. If you're doing it repeatedly and could automate it, the meta-question is why you haven't.

[DORA metrics] — named-only  
*DORA* (DevOps Research and Assessment) — four metrics for software delivery performance: deploy frequency, lead time for changes, change failure rate, mean time to recovery. Reach for these if you start running PULSE-like systems for other people, where delivery cadence becomes a question in its own right.

[On-call, alerting policies] — named-only  
*On-call rotation* and *alerting policies* — the social and organizational scaffolding for keeping a service responsive. Reach for these if you scale beyond personal use; for solo work they're overkill, though the underlying principle ("decide what's worth interrupting yourself for, in advance") translates.

## 6. Failure modes — the layer that names what goes wrong

This is where the vocabulary pays back the most. Most of these have distinct names because they have distinct shapes, and reaching for the wrong frame ("oh, the model is just hallucinating") buries the real diagnosis. Treat the names as differential diagnoses — when something is wrong, walk down the list and ask which one fits.

[Hallucination] — load-bearing  
The model produces plausible-sounding output that isn't grounded in reality (no source, no fact, no provided context supports it). Worth subdividing because the defenses differ. *Confabulation*: filling in gaps with made-up specifics that fit the pattern of the request — most common when the model lacks information and isn't allowed to say so. *Citation hallucination*: making up sources, references, file paths, function names. Especially common when asked for citations the model can't actually provide. *Capability hallucination*: claiming to have done something it didn't actually do (especially in agentic contexts — "I've updated the file" when no edit tool was called). *Persona hallucination*: drifting into invented backstory or context. The general defense is grounding (force the model to cite from provided context, then verify the citations). The specific defense for agentic systems is *verifiable side effects* — don't trust that the agent did something; check, by rereading the file or re-running the query.

[Prompt injection] — load-bearing  
User-controlled or document-controlled text that hijacks the model's instructions. Two flavors. *Direct injection*: a user message says "ignore previous instructions and do X." *Indirect injection*: text inside a document the model is asked to summarize contains instructions the model then follows. Indirect is the harder problem because you can't tell injection apart from legitimate document content — and the document might have been crafted by someone who anticipated your model would read it. The defenses are partial: input guardrails to scrub obvious patterns, careful prompt structure that distinguishes data from instructions (most reliably by *not* concatenating user content into the system prompt), and treating tool outputs as untrusted. The key insight: there is no perfect defense at the prompt level. Your defense has to be at the *capability* level — what damage can a successfully-injected prompt actually do? If the answer is "exfiltrate data, take destructive actions," tighten capabilities.

[Context decay, lost in the middle] — load-bearing  
LLMs degrade with long context. *Lost in the middle*: information placed in the middle of a long context is recalled worse than information at the start or end (a U-shaped attention curve, in many models). *Context decay* more generally: the longer the context, the more the model's behavior drifts from its instructions. This is one of the highest-leverage things to know about. The defense isn't "use a bigger context window" — it's structural. Put critical instructions at the start *and* end. Compress history when you can. For agent loops, summarize and prune rather than appending forever. PULSE's compression rules (Maps as indices, not containers; "read the Map only, follow links only when working") are exactly this defense applied at the system level. The recurring temptation: "the model should be able to handle this much context, the docs say it can." The model can; it just won't, reliably, and your evals should be measuring the degradation, not the marketing claim.

[Context poisoning] — load-bearing  
Once bad content (hallucinated tool output, injected instruction, misclassified Note) is in the context, subsequent reasoning is conditioned on it. A model that has hallucinated a function will keep using the hallucinated function. A multi-turn agent that misread a user's intent will keep refining its plan around the misread. The defense is to *catch and correct early*, before the bad context propagates, and to *limit propagation* — sub-agents should return narrow results to the parent, not full transcripts. The contrast with simple hallucination: hallucination is a single bad output, poisoning is the persistent bad state that follows when that output isn't caught. Most production "the agent went off the rails" stories are poisoning, not first-order hallucination.

[Sycophancy] — load-bearing  
The model agrees with the user's premise even when the premise is wrong, because the training pressure rewards agreement. Especially insidious in evaluation: an LLM-as-judge will score things higher if asked in a leading way. Especially insidious in dyad work: if you express a preference, the model leans toward it. The defenses: ask in neutral form, sometimes ask the inverse to check for consistency, give the model permission to disagree explicitly, and run pairwise comparisons rather than absolute ratings (sycophancy distorts absolute scores more than relative ones). The PULSE-specific concern: any system where the user reviews the agent's surfacing has a sycophancy risk loop — the agent learns to surface what the user said yes to, which may be what the user already agreed with rather than what they should have surfaced. Calibration logs are partial protection here only if the user keeps scoring honestly even when "this should have been different" is the harder answer.

[Drift] — load-bearing  
The system's behavior changes over time even though no one made an explicit change. Several distinct types worth distinguishing. *Model drift*: the underlying model was updated, and behavior changed underneath you. *Data drift*: the inputs your system sees have shifted (new kinds of efforts, new vocabulary, different tone). *Prompt drift*: you've made many small prompt edits, each justified, and the cumulative effect is something you didn't choose. *Behavior drift*: the system as a whole is producing different output distributions for the same kind of input. The detection is the same in all cases — you need a stable eval that you re-run periodically and compare against historical baselines. Without that, drift is invisible until it's bad. The cruel feature of drift: it's slow, so each individual change feels fine, and the human gets used to the new behavior incrementally. By the time it's obviously wrong, you can't easily say what changed.

[Cascading failure] — load-bearing  
One small failure causes a larger one, which causes a larger one. In AI systems the classic pattern is: a small classification error sends an item to the wrong sub-agent, which produces a confused response, which the parent agent treats as authoritative, which corrupts subsequent reasoning. The defense is the bulkhead pattern (isolate failures) plus circuit breakers (stop propagating when something seems off). The general principle: every step that consumes the output of another step is a potential amplifier. In multi-agent systems, the amplification can compound — three layers of agents each making a small error is enough to produce nonsense at the top. Watch the depth of your delegation chains.

[Reward hacking, specification gaming, Goodhart's law] — load-bearing  
The system optimizes for the *measure* in a way that diverges from the *thing the measure was trying to capture*. Three names for closely related things, worth keeping all three because they get used by different communities. *Reward hacking* is the RL term: the agent finds an unintended way to score points. *Specification gaming* is the broader version: the system satisfies the literal goal in an unintended way. *Goodhart's law* is the social-science version: "when a measure becomes a target, it ceases to be a good measure." All three apply directly to evals. If your eval rewards conciseness, the system will be terse to the point of unhelpfulness; if it rewards confidence, the system will be confidently wrong. The defenses are layered: multiple evals that capture different aspects (so gaming one doesn't pass the others); periodic human spot-checks; rotating the eval set; and most importantly, *awareness that every metric will eventually be gamed*, so design with that in mind. The deeper point: there's no "right" metric you can pick that escapes Goodhart — only a discipline of revising metrics as you notice them being gamed.

[Mode collapse] — load-bearing  
The system's outputs become repetitive — same phrasings, same structure, same conclusions for varied inputs. Different from drift in that it's a *loss of variety*, not a change of distribution. Common cause in personal AI work: heavy prompting toward a particular style, repeated training on a narrow eval, or low-temperature settings combined with a specific persona. The detection: look at the diversity of outputs over a varied input set. The defense: deliberately preserve variety (sample at higher temperature for some workflows, vary prompts, evaluate on diverse inputs). The named contrast: mode collapse is variety loss, *flaky behavior* is excess variety; both can be problems and the correct response depends on which you have.

[Reification] — load-bearing  
PULSE's word for the failure where a useful working frame hardens into fixed structure that's no longer being questioned. A label gets applied because it's available, then keeps getting applied because it's been applied before. A priority sticks because the system computed it once. A classification persists because no one re-examined it. Reification is the failure mode that makes long-running AI systems quietly worse over time, *even without any bug* — the system loses pliancy and becomes a museum of past judgments. The defense is architectural: build in places where things are *re-evaluated* rather than carried forward, and treat persistent labels as suspect by default. Sati, in PULSE's framing, is the meta-noticing of reification. The translation outside PULSE: any AI system that accumulates state needs explicit moves to *unlearn* — to question whether the accumulated assumptions are still earning their keep.

[Survivorship bias in evals] — load-bearing  
Your eval set contains the cases you remembered to add — usually cases where the system failed in ways you noticed. The cases where the system silently failed in ways you didn't notice are absent. So your evals over-represent the failures you've already learned to look for and under-represent novel failures. The mitigation: periodically sample real usage at random and add cases to the eval set whether or not you have a strong opinion about the right answer. If you only add cases when you already see what's wrong, you're only ever solving yesterday's problems. The harder mitigation: occasionally invite an outside viewpoint to look at recent outputs and tell you what's weird; the things they notice are usually things you've stopped seeing.

[Stale context, stale memory] — load-bearing  
The system reasons from outdated facts — a calibration that no longer matches reality, a Map state that hasn't been refreshed, a remembered detail that contradicts the current code. Especially common when the system has both "what it remembers" and "what's actually true now," and trusts the memory over the world. The defense: timestamps on everything, with explicit checks against the current state at decision points; and an instinct to *verify* memory before acting on it, especially when the memory makes a specific claim about something that could have changed. The PULSE convention "before recommending from memory, verify the file/symbol still exists" is exactly this discipline. The contrast with drift: drift is the system changing slowly without notice; staleness is the system holding on to a snapshot that was true once and isn't now.

[Flaky behavior, non-determinism] — load-bearing  
The same input produces different outputs across runs. Inherent to LLM systems (and not always bad — variation is also creativity). The failure mode is when flakiness is being treated as a bug to be eliminated through brute force (low temperature, rigid prompting) when actually the system needs to be designed to *tolerate* variation. A robust system gives acceptable answers across a range of model outputs; a fragile system requires the model to produce one specific output. Aim for robust. The named contrast with software flakiness: flaky tests in deterministic systems are bugs; flaky outputs in LLM systems are baseline behavior. If your design treats them as bugs, you'll over-engineer determinism that fights the substrate.

[Prompt brittleness] — adjacent  
Closely related to flakiness: small, semantically irrelevant changes to the prompt produce large changes in behavior. Reformatting the system prompt breaks the workflow; renaming a variable in the example changes the answer. Detection is adversarial eval (vary the prompt, check that behavior is stable). Defense is, among other things, redundancy (state critical instructions multiple ways), structure (use clear delimiters between sections), and not relying on subtle phrasing for important behaviors. The diagnostic question: if a colleague rephrased my prompt for clarity, would my system still work? If the answer is "probably not," you have brittleness baked in.

[Skill atrophy in the operator] — load-bearing  
Underrated and worth naming. Over time, a working AI system erodes the operator's ability to do the underlying work themselves — including the ability to notice when the system is wrong. PULSE knowing your priorities means you stop tracking them; an LLM judge means you stop developing judgment about quality. Not always bad (offloading is the point), but worth being deliberate. The defenses: occasionally do the work yourself without the system, periodically audit the system's outputs by hand, and hold onto the skills that let you notice when the system is degrading. The deeper concern: an operator with atrophied skill is also a less effective debugger when something does go wrong, so the system gets worse and the operator's ability to notice and fix it gets worse together — a slow joint failure mode.

[Alert fatigue, signal saturation] — adjacent  
If your monitoring fires too often, you stop responding. If your toil keeps growing because you keep adding measurements without removing them, you eventually drown. Both are second-order failures of having too many signals and too little attention. The discipline: fewer, higher-signal alerts; periodic pruning of measurements that turned out not to matter. The contrast worth keeping: noise isn't just failure to detect signal, it's active erosion of your ability to detect signal — a system with many false positives trains its operator to ignore the true positives.

[Confirmation bias in the human-in-the-loop] — load-bearing  
Specific to systems where a human reviews AI output: the human approves things that match expectation, rejects things that don't, and stops looking critically once a few approvals have established a pattern of trust. The result is a system that drifts in whatever direction the human's prior expectations point, without anyone noticing. Especially dangerous in evaluation contexts where you're judging whether a model output is good — you'll see what you expect to see. The defenses: blind evaluation (don't know which version produced which output until after scoring), structured review templates that force you to look at specific properties, and the discipline of writing down expected outcomes *before* seeing actual outcomes. The harder defense: actively look for evidence that contradicts your sense that the system is fine; positive evidence is too easy to find.

[Cargo-cult patterns] — adjacent  
The system pattern that worked elsewhere, copied without the conditions that made it work. Often shows up as "best practice" guardrails that don't apply, eval harnesses that test the wrong things because they were copied from a different domain, or runbooks that don't match the actual system. The defense is to ask what each pattern is *for* and whether the conditions hold here, before adopting it.

## Adjacent fields worth a look

**Cybernetics.** The original frame for thinking about feedback, control, and self-regulating systems — broader and more philosophical than control theory proper. Brings *requisite variety* (Ashby) and the *purposive system* lens (Wiener, Beer). Read it as the conceptual ancestor of everything in this reference, and as a useful corrective when the engineering vocabulary starts feeling mechanical. The Stafford Beer line worth carrying: "the purpose of a system is what it does" — a brutal way of cutting through stated intentions to look at actual behavior.

**Statistical process control (SPC).** From manufacturing quality, mid-20th century. Brings *control charts*, the distinction between *common cause variation* (natural noise in the system) and *special cause variation* (something specific happened), and the discipline of treating noise and signal differently before reacting. The core unlock: many "problems" in a noisy system are just normal variation, and reacting to them makes things worse — a lesson AI operators rediscover painfully when they tweak prompts in response to single bad outputs. *X-bar / R chart* is the canonical visual; the principles translate well to LLM eval scores plotted over time.

**Cognitive accident research (Reason, Rasmussen, Dekker).** From aviation and medicine. Brings the *Swiss cheese model* of layered defense, the *active vs latent failure* distinction, and the broader insight that human error is a symptom of system design more often than a cause of failure. Translates directly to thinking about AI failures, where the latent conditions (vague specifications, ambiguous tools, brittle prompts) are usually more important than the proximate trigger (model output that "should have been better").

**Software testing literature (Beck, Hunt, Crispin).** Brings the test pyramid (and its critiques), the discipline of *testing behavior not implementation*, the long debate over what "good test" means, and the practical wisdom that fast tests get run and slow tests don't. Concept-handles worth importing: *test as specification*, *given-when-then*, *the cost of slow tests*, *contract test*. Especially useful as a corrective to the eval-set-as-property-test view, which can drift toward over-specification.

**MLOps / ML observability.** Closer cousin: same problem space, narrower focus on traditional ML systems. Brings *training-serving skew*, *concept drift* (more rigorously defined than the LLM version), feature stores, and a more developed vocabulary around lineage and reproducibility. Translates partially to LLM systems, with the caveat that some assumptions (a model is a fixed artifact you trained, you can retrain it on new data) don't hold for foundation models you're consuming as a service.

**Reliability engineering proper (originally hardware, then SRE).** Brings the formal concepts of *MTBF* (mean time between failures), *MTTR* (mean time to recovery), *FMEA* (failure mode and effects analysis), and the discipline of explicit reliability targets. The FMEA exercise — systematically enumerate possible failures and ask, for each, what would catch it — translates very well to AI systems and is rarely done; running one on your own AI workflow tends to be uncomfortable in a useful way.

## Starter projects

**A regression set for `/pulse`.**  
Curate ten to twenty real session states (snapshots of vault state plus a "what should /pulse have surfaced here" answer) and write a small harness that runs `/pulse` over each state and grades the result against your expected surfacing. Exercises *eval*, *golden set*, *regression suite*, *decision trace* (you'll want session log output to debug failures). Most likely to first surface: *eval rot* — you'll discover that "what should have been surfaced" was less stable than you thought, and that your own grading is itself a moving target. That discovery is the real lesson; the harness is the artifact.

**A drift detector for priority weights.**  
Log the daily ranked list from `pulse-calc.py` and build a small comparison that flags when the top-10 changes in ways larger than typical day-over-day variation. Exercises *drift*, *control chart* discipline (what's the natural variation, vs what's an anomaly), *common vs special cause*. Most likely to first surface: *flaky behavior* misread as drift — you'll need to define "typical variation" carefully or every day will look anomalous, and the calibration of your "normal" envelope is the real work.

**An LLM-as-judge for `/triage` classifications.**  
Save the last N triaged items along with which Map they were assigned to. Have a fresh judge (different model or different prompt structure) re-classify them blind and compare. Exercises *LLM-as-judge*, *calibration*, *pairwise scoring*, and the specific failure mode of *correlated judges*. Most likely to first surface: *sycophancy* — if you reveal the original classification to the judge, it will agree with it; you'll learn to actually run blind, which is more procedural friction than it sounds.

**A blameless-postmortem template for "PULSE missed surfacing X."**  
When you notice the system buried something important, write it up: timeline, contributing factors (multiple, named), what's changing as a result. Do it five times over a month. Exercises *postmortem*, *Swiss cheese model*, *active vs latent failure*. Most likely to first surface: the difficulty of resisting *single-cause stories* — your first instinct will be "the weight algorithm is wrong" when usually three or four conditions had to align for the surfacing to fail. Watch for the moment you stop the analysis before reaching the latent causes.

**A canary harness for skill changes.**  
Before changing a skill (say, `/pulse`), capture its output on a small fixed set of inputs. After the change, re-run on the same inputs and diff. Exercises *canary*, *snapshot test*, *circuit breaker* (treat large diffs as a "stop and investigate" signal). Most likely to first surface: *over-specification* — your snapshot tests will fail on legitimate improvements, and you'll either discard them in frustration or learn the discipline of property-style assertions that catch real regressions without flagging cosmetic changes.
